Prepare OpenTelemetry Java agent tracing assets for a Java project so Bootstrap can trace request and message flow without custom probes.

DEFAULTS:
- If `projectDir` is omitted, use `${directory}/mocknet` when it exists, otherwise `${directory}`
- If `command` is omitted, Maven projects default to `mvn spring-boot:run`
- If `agentJarPath` is omitted, download the official OpenTelemetry Java agent into `.bootstrap/otel/<serviceName>/`

OUTPUTS:
- `otel.env` with OpenTelemetry environment variables
- `run-with-otel.sh` launcher for the Java app
- `stop-app.sh` helper to stop the managed app
- `app-status.sh` helper to report whether the managed app is running
- `start-jaeger.sh` helper for local Jaeger
- `stop-jaeger.sh` helper to stop Jaeger

WHEN PREPARING OR VALIDATING TRACING:
- Prefer launching the app through `run-with-otel.sh` so the tool-controlled `otel.env` is actually applied.
- Do not stop at Jaeger or OTLP health alone. Exporter health is necessary but not sufficient.
- After startup, send known-good application traffic and verify exact business/component span names in Jaeger before declaring success.
- For `mocknet`, the key business spans are:
  - `TradeSubmissionController.submitTrade`
  - `TradeIngestionService.processTradeXml`
  - `TradeMatchingEngine.processMatchingMessage`
  - `NettingCalculator.processNettingMessage`
  - `TwoPhaseCommitCoordinator.executeTransaction`
- Do not treat `ps` output alone as proof that the Java agent is missing when `JAVA_TOOL_OPTIONS` may be injecting `-javaagent`.
- Do not treat `span.serviceName` being absent in Jaeger as proof that service attribution is broken; inspect the trace `processes` map.
- Do not declare tracing healthy if Jaeger only shows queue polling spans, SQL `SELECT` spans, or `Transaction.commit`.

USE THIS TOOL WHEN:
- You need end-to-end trace visibility for a Java service
- You want Bootstrap to prepare tracing against `mocknet/`
- You want a repeatable local Jaeger setup with minimal custom code
